## CRITICAL findings

### C1. Daily price refresh has not run since 2026-05-19 (11 days stale)

**Evidence** (from `_audit_data.py`, `_audit_data2.py`):

```
2026-05-19: 2,296 rows refreshed  ← last full universe refresh
2026-05-20 .. 2026-05-27: 14 rows/day  (only AAPL/NVDA/SLGL + 11 SPDR ETFs)
2026-05-28: 11 rows  (only the 11 SPDR ETFs from warm_sector_etfs.py)
```

The 14-row days correspond to one-off ticker fetches during research
work, not the full ~2,300-ticker universe `daily_price_refresh.py` is
supposed to maintain.

**Impact**: MTM for stock sleeves uses 11–23 day old prices for
197 / 204 open holdings:

| Sleeve | Stale holdings | Latest cached |
|---|---|---|
| mom_v1_paper | 99 / 100 | 2026-05-07 or 05-19 |
| mom_v2_paper | 49 / 50 | 2026-05-07 or 05-19 |
| mom_roa_6535_paper | 49 / 50 | 2026-05-07 or 05-19 |
| sector_top4_paper | 0 / 4 | 2026-05-28 |

The dashboard "+/-X.XX% vs start" numbers reflect performance through
~May 7/19, not actual current state. User has been reading stale numbers
for ~11 days.

**Root cause**: No Windows scheduled task installed. `daily.bat` requires
manual invocation. User likely hasn't been running it.

**Fix recommendation** (in order):

1. **Run `daily_price_refresh.py` now** to bring cache up to date
   (~5–8 min for full universe).
2. **Install a Windows scheduled task** that runs `daily.bat` at
   ~17:00 ET each weekday (after market close).
3. **Add a freshness check to `paper_mtm.py`**: WARN loudly if median
   holding price age > 3 days. Fail loudly if > 7 days.

### C2. Empty / shrunken universe = catastrophic rebalance risk

**Evidence** (from `_audit_rebal.py`):

| as_of | universe size |
|---|---|
| 2026-05-19 | 1,472 |
| 2026-05-28 | 11 (only SPDR ETFs survive freshness filter) |
| 2026-05-30 | 0 (weekend + stale) |

`paper_rebalance.py` correctly aborts when universe is **empty** (line
92-94), but **silently proceeds** with a tiny universe. If the user ran
the monthly rebalance on 2026-05-28 with current stale data:

- `mom_v2_paper` would SELL all 50 stock holdings → BUY 11 ETFs.
- Same for `mom_v1_paper` (top-100 from 11-ticker universe = whatever
  ranks; would still liquidate stocks).

This would happen silently. The monthly rebalance is scheduled for
2026-06-01 (Monday). If the user runs `rebalance.bat` before fixing C1,
this will fire.

**Fix recommendation**:

Add a minimum-universe-size guard to `paper_rebalance.py`:

```python
MIN_UNIVERSE = {"sector_top4_paper": 11}  # exact for ETF universe
DEFAULT_MIN = lambda top_n: max(top_n * 2, 200)

required = MIN_UNIVERSE.get(strategy_name) or DEFAULT_MIN(top_n)
if len(universe) < required:
    log.error("Universe size %d < required %d for %s. Aborting.",
              len(universe), required, strategy_name)
    return 0
```

## HIGH findings

### H1. Dashboard "Stale" flag is broken

**Location**: `trading_bot/dashboard/web.py:1868`

```python
stale = p["ticker"] not in last_closes
```

Only flags tickers with **no** cached price. Doesn't flag tickers with
**old** cached prices. Effective result: dashboard shows 99/100 mom_v1
holdings as "fresh" while their prices are 11–23 days old.

**Fix**: store the date alongside the price in `last_closes`, and flag
stale if `date_today - last_date > N days`.

## MEDIUM findings

### M1. `_load_paper_state` SQL is non-standard

**Location**: `trading_bot/dashboard/web.py:1738-1744`

```sql
SELECT ticker, MAX(key_date) AS d, price
FROM price_cache WHERE kind='close' AND ticker IN (...)
GROUP BY ticker
```

`price` is unaggregated and not in GROUP BY. Modern SQLite returns the
price from the row matching MAX(key_date) (verified), but this relies
on the SQLite-3.7.11+ "bare column with MIN/MAX" behavior, not standard
SQL. Fragile to a DB-engine swap.

**Fix**: use a correlated subquery or window function.

### M2. `paper_mtm.py` does not warn on aged prices

**Location**: `scripts/momentum/paper_mtm.py:43-52`

Only counts a price as "stale" if missing entirely. Otherwise silently
uses the most recent close, however old. Combined with C1, this is why
the user has no signal that the cache went stale.

**Fix**: track price age in `compute_nav()`, return median/max age, and
log a warning if exceeding a threshold (e.g., 3 days).

### M3. Universe filter on date `D` requires a close ON `D`

**Location**: `trading_bot/factors/universe.py:155-160`

Strict same-day match. Combined with `daily_price_refresh` missing days,
universe shrinks rapidly as stale tickers age past today. This is by
design (matches backtest semantics) but pairs badly with C1.

**Fix**: tolerate 1-2 trading-day staleness in `tradeable_universe`
when `as_of >= today - 2`. This is the difference between rebalancing
"on Monday with stale weekend data" working vs aborting.
