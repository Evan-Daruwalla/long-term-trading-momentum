## C1 (CRITICAL, fixed): daily_price_refresh wrote dividend-ADJUSTED closes
into a dividend-UNADJUSTED cache

- The cache convention everywhere (scripts/shared/warm_*, all 7
  market_data.py fetch-on-miss sites, warm_sector_etfs, warm_vol_letf_etfs)
  is `auto_adjust=False`: split-adjusted, dividend-unadjusted closes.
- `daily_price_refresh.py` alone used `auto_adjust=True` — every daily run
  since 2026-05-30 stamped dividend-adjusted values over the trailing 30 days
  of ALL ~4,300 tickers.
- Mechanism: adjusted differs from unadjusted only for dates BEFORE an ex-div,
  so each name with an ex-div inside the trailing window got its pre-ex-div
  rows scaled down by the dividend (~0.5-1%/event).
- **Measured** (pre-repair): XOM 11 rows -0.67%, JNJ 17 rows -0.57%, CVX 13
  rows -0.91%, IBM 6 rows -0.73% in 2026-04-30→06-09. Non-payers + names with
  no recent ex-div clean. Extrapolated: thousands of rows across div payers.
- Impact: small (momentum top-50s are mostly non-payers; frozen-test windows
  predate the refresh region and passed throughout) but compounding daily,
  and it biased ranks/MTM slightly against div payers near ex-div.
- **Fix**: refresh flipped to `auto_adjust=False` (+ convention documented in
  code and CLAUDE.md); seam repaired by re-running refresh `--days 60`
  (217,438 rows rewritten); repair verified against recorded bad values
  (XOM/JNJ/CVX/IBM all back to true unadjusted closes); frozen tests re-run
  → all 4 at 0.0000pp; all 8 sleeves re-MTM'd on repaired data.
- Residual risk: paper_nav rows written 05-30→06-08 used the then-mixed data
  (≤~1% error on div-payer holdings). Left as historical record — same
  treatment as entry prices, which are booked fills.
