## 1. OBJECTIVE

Harden the paper-trading system so its track record stays trustworthy while running unattended.

The project is Evan's paper-trading systematic-equity platform: 11 live sleeves (the "07-06
cohort", all re-inceptioned at $100k on 2026-07-06 with broker-realistic fills), three of them
mirrored to real Alpaca PAPER accounts, a daily 5:15pm mark-to-market task, and a monthly
rebalance task that fires **unattended** starting 2026-08-01. The track record — clean,
verifiable, never backdated, never fabricated — is the asset; Evan is 17 and building the rigor
and the record before he can trade real money at 18.

The system just crossed from "operated by hand" to "runs by itself." This plan builds the
guardrails, verifications, and reports that let an unattended system be trusted: data-quality
gates, post-run verification, experiment-integrity tracking, backups, and slippage measurement —
plus bringing the stale documentation back in sync. The user is Evan reviewing dashboards and
reports; the outcome is that a silent failure or bad data can no longer quietly contaminate the
record.

### Known failure classes (all real, all from the record — the guardrails in M2–M3 target these)

1. **Incomplete yfinance publication** (Appendix AU): a trading day's closes arrive for only
   ~4,400 of ~5,200 tickers and never settle that evening. The 07-06 deploy used a manual
   "coverage ≥ 5,000 closes" gate; nothing enforces this on the recurring tasks.
2. **Split misapplication** (KLAC, 2026-06-12): yfinance applies a split to history days early →
   bogus >1000% single-day moves → wrong ranks/NAVs. The tell is a giant one-day move.
3. **Cache gaps** (the ~815-ticker 2019-2025 hole found 2026-06-13): silent missing history
   phantom-ranks stale names. Cost weeks of contaminated live record.
4. **Pre-inception NAV pollution** (Appendices AU/AV): the daily MTM writes $100k rows for
   sleeves whose inception hasn't arrived (holiday weekends), which then need manual deletion.
5. **Friday-spike / bogus-close corruption** (2026-05-28 audit): mitigated by
   `MAX_HIST_RATIO=100` in the refresh path, but detection is passive.

### Must not break

- **Frozen regression tests**: `trading_bot/strategies/test_strategies.py` — 4 pinned configs
  must stay at d=±0.0000pp after ANY change. Run them after every task that touches Python.
- **price_cache convention**: closes are split-adjusted, dividend-UNadjusted
  (`auto_adjust=False`). Every writer honors it; never add one that doesn't.
- **Never run concurrent `factor_backtest` processes on the same DB** (silent corruption).
  This plan shouldn't need factor_backtest at all.
- The live decision record: LLM decisions are never backdated, NAV history is never rewritten
  (deleting *pre-inception* junk rows per an explicit task is the only sanctioned deletion).
- The scheduled tasks and their timing; `alpaca_keys.env` (never print, commit, or move it).
- `daily_report.md` ordering (newest-last) and both HTML twins (script-generated only).

## 4. CONSTRAINTS

- **Ops/infra only** — see the scope guard at the top. Out of scope: new strategies or sleeves,
  factor/parameter changes, universe changes, backtests, changing any LLM prompt or decision
  rule, un-deferring anything Evan deferred, renaming scheduled tasks, "cleaning up" the DB.
- **Read-only by default against `var/trades.db`.** New checks/reports open the DB read-only
  (`sqlite3` URI `file:...?mode=ro`). The only write-path changes allowed are the ones named in
  tasks (M3.1 inception guard, M5 backups, M6 slippage_log appends).
- **Write-path changes are tested on a copy, never live-first**: copy `var/trades.db` to the
  scratch area (check ≥10 GB free disk first) or build a small fixture DB; only after tests pass
  does the change run against live.
- **Frozen tests after every Python change**: `.venv\Scripts\python.exe -m pytest
  trading_bot/strategies/test_strategies.py` (or the invocation the file documents) — must be
  d=±0.0000pp. Report the actual output, never "should pass".
- **NEVER fabricate data, fills, prices, or results.** Missing data is reported as missing.
  This is a standing order with zero exceptions.
- **Don't run anything that trades**: no `paper_rebalance`, no `*_ops rebalance/decide`, no
  `alpaca_sync --execute` — those belong to the scheduled tasks and Evan. Dry-run/read modes
  only, and only where a task says so.
- **Timing**: avoid running anything DB-heavy between 5:00–6:30pm local (daily MTM window) or
  on the 1st trading day of a month (rebalance). Prefer mornings.
- **Windows**: PowerShell 5.1 (no `&&`); batch-file edits must stay pure ASCII (one non-ASCII
  byte silently corrupts cmd.exe parsing — this bug already happened, Appendix AS); never
  rewrite data files with PowerShell.
- **Commits**: this document authorizes a local commit after each completed task. Never push
  without Evan's instruction.
- **Docs cadence**: record entry (absolute dates, append-only) per task; HANDOFF/state per
  milestone; regenerate HTML twins by script only.

### M2 — Data-quality guardrails

1. **Coverage gate script.** New `scripts/momentum/check_coverage.py`: read-only query of
   `price_cache` for the latest trading date's close count vs a baseline (rolling median of the
   prior 10 trading days); exit 0 if ≥ floor (default: max(5000, 90% of baseline), overridable
   via `--floor`), exit 1 with a clear log line otherwise. Mirror the manual gate used in the
   07-06 deploy (Appendix AV: 5,206 vs ~5,200 baseline, 5,000 abort floor). Test against the
   live DB read-only (it's a query, not a write). Done: script returns 0 today with the count
   printed, and returns 1 when given an absurd `--floor 999999`.
2. **Wire the gate into `daily.bat`.** After the price refresh step and BEFORE MTM: run
   check_coverage; on failure, skip MTM, write a `COVERAGE FAIL` line into
   `var/last_daily_run.log`, and still exit nonzero so the task history shows failure. Keep the
   .bat pure ASCII; test by running `daily.bat` manually on a completed trading day (read the
   script first — confirm rerunning MTM same-day is idempotent, which existing usage implies;
   if unsure, test MTM idempotency on a DB copy first). Done: manual run passes the gate and
   completes; a forced-floor failure run skips MTM and logs the FAIL line.
3. **Anomaly detector.** New `scripts/momentum/check_anomalies.py` (read-only): for the latest
   close date, flag any ticker in (a) current positions of any sleeve or (b) the price_cache
   whose 1-day move exceeds a threshold (default 300% for held names, 1000% cache-wide — the
   KLAC tell), plus any held ticker with a missing latest close. Output: console + append a
   dated section to `var/anomaly_report.log`. Wire it into `daily.bat` after MTM as
   non-blocking (report, don't halt). Done: clean run on live data produces an explicit
   "0 anomalies" dated entry; frozen tests still ±0.0000pp.
4. **Cache-gap audit.** New `scripts/momentum/check_cache_gaps.py` (read-only): for every
   ticker currently rankable (has recent closes), verify history continuity over the lookback
   the strategies actually use (12-1 momentum needs ~13 months) — flag tickers with holes
   > N consecutive trading days (default 5) inside their own first-to-last date range. This is
   the 815-ticker-class detector. One full run recorded to `var/cache_gap_report.log` +
   record entry summarizing findings honestly (finding real gaps is a REPORT, not a license to
   backfill — backfilling is Evan's call). Done: full run completes; findings documented.

**Gotchas that will bite you:**

- The frozen tests are the contract. `trading_bot/strategies/test_strategies.py` must print
  d=±0.0000pp after every change, even ones that "obviously can't affect strategies". Run it
  every time and paste the real output into the record.
- `price_cache` is split-adjusted, dividend-UNadjusted (`auto_adjust=False`). Any comparison to
  external prices (Alpaca fills!) must account for this — fills are raw traded prices, which is
  fine for slippage (same-day), but never "correct" the cache to match a vendor.
- The DB is live and ~5 GB. Open read-only unless the task explicitly writes. Never two writers
  at once. Stay out of the 5:00–6:30pm window and the 1st trading day of the month.
- Batch files: cmd.exe parsing silently corrupts on a single non-ASCII byte (already happened,
  Appendix AS). Edit .bat files in pure ASCII, test by running them, and remember a stray root
  file named like a builtin can shadow commands.
- The scheduled task named `monthy-llm-rebalance` is really named that (typo included). Renaming
  it breaks the automation chain — leave it.
- `alpaca_keys.env` holds live API keys: never print, log, commit, or relocate it.
- Sleeve NAV/decision history is sacred. The only sanctioned write to history in this whole plan
  is the M3.1 guard *preventing* bad rows. If you find data that looks wrong, REPORT it in the
  record; deleting/fixing is Evan's call.
- The record and daily_report have script-generated HTML twins — regenerate via
  `scripts.render_record_html` / `render_daily_report_html.py`, never hand-edit the HTML.
  daily_report.md is newest-LAST (opposite of intuition).
- "DO NOT MAKE ANYTHING UP" is the project's standing order. A report with honest gaps beats a
  complete-looking one with invented numbers, every time. If a query returns nothing, the
  deliverable is "no data", not an estimate.
